## Overview

The Purchase Orders feature (Guía y Orden de Compra) allows you to create, track, and manage purchase orders for goods and services. Each purchase order records the supplier, purchase date, item description, quantity, and price, serving as the foundation for asset management and financial tracking.

<Info>
  Purchase orders are stored in the `grr-ordendecompra` database table. Assets registered in the system must reference a valid purchase order code.
</Info>

## Creating a Purchase Order

<Steps>
  <Step title="Access Purchase Orders">
    Navigate to the Guia page from the main menu. You'll see a list of all existing purchase orders.
  </Step>
  
  <Step title="Open Registration Form">
    Click the **Registrar** button to open the purchase order creation modal.
  </Step>
  
  <Step title="Enter Purchase Order Details">
    Fill in all required fields:
    
    <ParamField path="nombre" type="string" required>
      **Supplier Name (Proveedor)**: Name of the supplier fulfilling this order
      - Must be 4-20 characters
      - Only letters allowed (including Spanish characters)
      - Should match a supplier in your supplier directory
      - Example: "Comercial Lopez", "Distribuidora SAC"
    </ParamField>
    
    <ParamField path="fecha" type="date" required>
      **Purchase Date (Fecha Compra)**: Date the purchase order was created
      - Format: YYYY-MM-DD
      - Example: 2026-03-04
    </ParamField>
    
    <ParamField path="descripcion" type="string" required>
      **Description (Descripción)**: Description of items being purchased
      - Must be 4-20 characters
      - Only letters allowed
      - Example: "Equipos oficina", "Laptops Dell"
    </ParamField>
    
    <ParamField path="cantidad" type="integer" required>
      **Quantity (Cantidad)**: Number of items being purchased
      - Must be a positive integer
      - Example: 5, 10, 25
    </ParamField>
    
    <ParamField path="precio" type="number" required>
      **Price (Precio)**: Total price or unit price for the purchase
      - Must be numeric
      - Stored as Double precision
      - Example: 1500.50, 25000
    </ParamField>
  </Step>
  
  <Step title="Submit the Form">
    Click **Grabar** to create the purchase order. The system will:
    - Validate all fields meet the requirements
    - Insert the record into the database with an auto-generated code
    - Display a success confirmation message
    - Refresh the purchase order list
  </Step>
</Steps>

<Note>
  The purchase order code (`codigo`) is automatically generated by the database and serves as the unique identifier for linking assets to this purchase order.
</Note>

## Field Reference

Purchase orders contain the following information:

| Database Field | Display Name | Type | Description |
|----------------|--------------|------|-------------|
| `codigo` | Código | Integer | Auto-generated unique purchase order code |
| `proveedor` | Proveedor | String | Supplier name (4-20 characters) |
| `fecha_compra` | Fecha Compra | Date | Purchase order date (YYYY-MM-DD) |
| `descripcion` | Descripción | String | Item description (4-20 characters) |
| `cantidad` | Cantidad | String | Quantity of items |
| `precio` | Precio | Double | Purchase price |

<Warning>
  The `cantidad` field is stored as String in the entity but validated as an integer. This allows flexibility but requires proper input validation.
</Warning>

## Editing a Purchase Order

<Steps>
  <Step title="Locate the Purchase Order">
    Use the DataTable search to find the purchase order by code, supplier, or description.
  </Step>
  
  <Step title="Open Edit Modal">
    Click the green **Editar** button on the purchase order's row. The system loads current data via AJAX from `ServletGuiaJSON`.
  </Step>
  
  <Step title="Modify Information">
    Update any fields as needed:
    - All validation rules still apply
    - The purchase order code cannot be changed (read-only)
    - Date must follow YYYY-MM-DD format
  </Step>
  
  <Step title="Save Changes">
    Click **Grabar** to update the purchase order. The system executes:
    ```sql
    UPDATE gestionbienes.`grr-ordendecompra` 
    SET proveedor=?, fecha_compra=?, descripcion=?, cantidad=?, precio=? 
    WHERE codigo=?
    ```
  </Step>
</Steps>

## Deleting a Purchase Order

<Warning>
  **Critical**: Before deleting a purchase order, verify that no assets are linked to it. Deleting a purchase order that has associated assets may cause data integrity issues.
</Warning>

<Steps>
  <Step title="Select Purchase Order">
    Locate the purchase order in the table listing.
  </Step>
  
  <Step title="Initiate Deletion">
    Click the red **Eliminar** button on the purchase order's row.
  </Step>
  
  <Step title="Confirm Deletion">
    A confirmation modal appears with "Seguro de eliminar?":
    - Click **SI** to confirm deletion
    - Click **NO** to cancel
  </Step>
  
  <Step title="Complete Deletion">
    The system removes the record:
    ```sql
    DELETE FROM gestionbienes.`grr-ordendecompra` WHERE codigo=?
    ```
  </Step>
</Steps>

## Validation Rules

<Accordion title="Field Validation Patterns">
  ### Supplier Name (Proveedor)
  - **Pattern**: `^[a-zA-ZñÑ\s\áéíóúÁÉÍÓÚ]{4,20}$`
  - **Length**: 4-20 characters
  - **Characters**: Letters (including Spanish), spaces
  - **Error**: "Campo proveedor MIN:4 - MAX:20"
  
  ### Purchase Date (Fecha)
  - **Pattern**: `^([0-9]{4})-([0-9]{1,2})-([0-9]{1,2})$`
  - **Format**: YYYY-MM-DD
  - **Error**: "Campo fecha es obligatorio yyyy-mm-dd"
  
  ### Description (Descripción)
  - **Pattern**: `^[a-zA-ZñÑ\s\áéíóúÁÉÍÓÚ]{4,20}$`
  - **Length**: 4-20 characters
  - **Characters**: Letters only
  - **Error**: "Campo descripción solo letras MIN:4 - MAX:20"
  
  ### Quantity (Cantidad)
  - **Pattern**: `^\d+$`
  - **Type**: Positive integer
  - **Error**: "Campo cantidad debe ser #entero positivo"
  
  ### Price (Precio)
  - **Pattern**: `^\d+$`
  - **Note**: Validation only checks for numeric digits (may need enhancement for decimals)
  - **Error**: "Campo precio debe ser numerico"
</Accordion>

## Business Rules

<Accordion title="Purchase Order Workflow">
  ### Creation
  1. Verify supplier exists in supplier directory
  2. Create purchase order with all required details
  3. System generates unique purchase order code
  4. Purchase order is now available for asset linking
  
  ### Asset Linking
  - Assets reference purchase orders via the `codigo_ordencompra` field
  - Multiple assets can reference the same purchase order
  - The link enables traceability from procurement to inventory
  
  ### Modification
  - Purchase orders can be updated at any time
  - Code cannot be changed to maintain referential integrity
  - Consider impact on linked assets when modifying descriptions
  
  ### Deletion
  - Only delete purchase orders with no linked assets
  - Deletion is permanent and immediate
  - Consider archiving instead of deleting for audit purposes
</Accordion>

<Accordion title="Data Integrity">
  - Purchase order codes are auto-increment integers
  - Codes serve as foreign keys in the `bienes` table
  - The `codigo_ordencompra` field in assets must match a valid purchase order code
  - Database enforces referential integrity through foreign key constraints
</Accordion>

## Database Operations

The system performs these SQL operations:

```sql
-- Create new purchase order
INSERT INTO gestionbienes.`grr-ordendecompra` 
VALUES(null, ?, ?, ?, ?, ?)

-- Update existing purchase order
UPDATE gestionbienes.`grr-ordendecompra` 
SET proveedor=?, fecha_compra=?, descripcion=?, cantidad=?, precio=? 
WHERE codigo=?

-- Delete purchase order
DELETE FROM gestionbienes.`grr-ordendecompra` WHERE codigo=?

-- Find by code
SELECT * FROM gestionbienes.`grr-ordendecompra` WHERE codigo=?

-- List all purchase orders
SELECT * FROM gestionbienes.`grr-ordendecompra`
```

## Integration with Asset Management

Purchase orders and assets are tightly integrated:

<Steps>
  <Step title="Create Purchase Order">
    First, create a purchase order for goods being acquired.
  </Step>
  
  <Step title="Note the Purchase Order Code">
    The system generates a unique code (e.g., 1, 2, 3, etc.).
  </Step>
  
  <Step title="Register Assets">
    When assets arrive, register them in Asset Management using the purchase order code.
  </Step>
  
  <Step title="Query Assets by Purchase Order">
    Use the "Consultar Bien por Orden de Compra" feature to view all assets linked to a purchase order.
  </Step>
</Steps>

<Info>
  The purchase order code creates an audit trail from procurement documentation to physical assets, supporting compliance and financial tracking requirements.
</Info>

## Best Practices

<Note>
  ### Recommended Guidelines
  
  **Before Creating Purchase Orders**
  - Ensure supplier exists in the supplier directory
  - Verify purchase date is accurate
  - Use clear, descriptive item descriptions
  
  **During Asset Receipt**
  - Match incoming assets to purchase order codes
  - Verify quantities received match purchase order quantities
  - Record actual arrival dates in asset management
  
  **For Audit and Compliance**
  - Retain all purchase orders for audit trail
  - Use the query feature to verify all assets have valid purchase orders
  - Consider implementing approval workflows before purchase order creation
</Note>

## Technical Implementation

<Info>
  **For Developers**: Purchase orders are managed through:
  - Entity: `com.bienes.entidad.GuiayOrdendeCompra`
  - DAO Interface: `com.bienes.interfaces.GuiaDAO`
  - DAO Implementation: `com.bienes.dao.MySqlGuiaDAO`
  - Servlet: `com.bienes.controlador.ServletGuia`
  - Service: `com.bienes.services.GuiaServices`
  - JSP View: `/Guia.jsp`
  - JSON APIs: `ServletGuiaJSON` (single record), `ServletGuiaListaJSON` (list for dropdowns)
  
  The table name uses backticks due to the hyphen: `` `grr-ordendecompra` ``
  
  Asset linkage uses a JOIN query in `MySqlBienesDAO.listarBienesporCodigodeComptra()` method.
</Info>
