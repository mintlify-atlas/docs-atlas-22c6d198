## DAO Pattern (Data Access Object)

The DAO pattern abstracts and encapsulates all access to the data source, providing a clean separation between business logic and data access logic.

        }
      }

## Additional Patterns

### Template Method Pattern (JDBC Operations)

<Accordion title="JDBC Template Pattern">
  All DAO methods follow a consistent template for database operations:
  
  ```java
  public int save(Entity bean) {
    int salida = -1;
    Connection cn = null;
    PreparedStatement pstm = null;
    try {
      // 1. Get connection
      cn = MySqlConexion.getConectar();
      // 2. Prepare SQL
      String sql = "insert into table values(...)";
      // 3. Create statement
      pstm = cn.prepareStatement(sql);
      // 4. Set parameters
      pstm.setString(1, bean.getField());
      // 5. Execute
      salida = pstm.executeUpdate();
    } catch (SQLException e) {
      e.printStackTrace();
    } finally {
      // 6. Clean up resources
      try {
        if(pstm != null) pstm.close();
        if(cn != null) cn.close();
      } catch (SQLException e2) {
        e2.printStackTrace();
      }
    }
    return salida;
  }
  ```
  
  This template ensures consistent error handling and resource management across all database operations.
</Accordion>

## Best Practices Observed

- **Interface-based programming**: DAOs are accessed through interfaces
- **Factory creation**: Objects are created through factories, not direct instantiation
- **Layered architecture**: Clear separation between presentation, business, and data layers
- **Resource management**: Proper cleanup of database resources in finally blocks
- **Consistent naming**: Clear naming conventions (MySql prefix for implementations)
- **Annotation-based configuration**: Use of `@WebServlet` instead of XML configuration

<Note>
These patterns work together to create a robust, maintainable architecture. Understanding how they interact is key to effective development and maintenance of the system.
</Note>
